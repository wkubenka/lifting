# Dumbbell Exercise Library Overhaul — Design

**Date:** 2026-04-28
**Status:** Approved by user, ready for plan-writing

## Context

The user (sole user of this personal Android app) decided the existing 800+ exercise library is not what they want. They've chosen a curated list of ~74 dumbbell exercises sourced from <https://www.endomondo.com/exercise/the-best-dumbbell-exercises>, each with a name and a one-sentence description. The Endomondo source has no images, and the user is fine working with names + descriptions only — pictures should be removed entirely. Existing workout history, personal records, and active workouts are not worth preserving; a clean slate is acceptable.

The intended outcome: workouts generated by the app pull from a smaller, dumbbell-focused library that the user actually wants to use, the exercise detail screen is text-only, and the APK is leaner without the bundled image assets.

## Scope (in)

1. Replace `app/src/main/assets/exercises.json` with the 74 Endomondo dumbbell exercises, conforming to the existing `ExerciseJson` schema (minus `images`).
2. Drop the `images: List<String>` field from `ExerciseEntity`, `ExerciseJson`, and the `DatabaseSeeder.toEntity()` mapper.
3. Bump Room DB to v7 with a `MIGRATION_6_7` that wipes per-exercise tables (so the seeder repopulates from new JSON) and drops the `images` column on `exercises`.
4. Remove image rendering from `ExerciseDetailScreen.kt` (`ExerciseImagePager`, `ImagePlaceholder`, related imports and the conditional that calls them).
5. Delete the `app/src/main/assets/exercises/` asset folder.
6. Remove the `coil3` dependency from `app/build.gradle.kts` and `gradle/libs.versions.toml` (no other consumer).
7. Update four test fixtures that build `ExerciseEntity` directly to drop the `images` parameter.

## Scope (out)

- No change to the 7-`MuscleGroup` taxonomy or `MuscleGroup.datasetMuscles` strings.
- No change to `MuscleGroupScorer` weights, the `WorkoutGenerator` selection logic, or the `IWorkoutRepository` interface.
- No change to `RecoveryConfigEntity` defaults or `UserPreferencesEntity`.
- No new exercise-level features (descriptions stay as `instructions[0]`, not a separate field).
- No simplified schema refactor (rejected option B in Q2 — the dataset-muscle bucketing stays as-is).

## Design

### 1. Data file & content (`app/src/main/assets/exercises.json`)

Full replacement of the JSON with 74 entries, each shaped like:

```json
{
  "id": "Dumbbell_Bench_Press",
  "name": "Dumbbell Bench Press",
  "force": "push",
  "level": "intermediate",
  "mechanic": "compound",
  "equipment": "dumbbell",
  "category": "strength",
  "primaryMuscles": ["chest"],
  "secondaryMuscles": [],
  "instructions": ["Lying on a bench, hold a pair of dumbbells on your thighs with neutral grip, then press upward to build chest, shoulders, and triceps."],
  "volumeMultiplier": 2
}
```

Conventions for filling in fields the Endomondo source doesn't provide:

- **`id`**: exercise name, spaces → underscores, special characters stripped.
- **`equipment`**: `"dumbbell"` for all 74. Renegade rows etc. count.
- **`category`**: `"strength"` for all.
- **`level`**: `"intermediate"` for all (user can adjust later via DB).
- **`secondaryMuscles`**: `[]` for all.
- **`volumeMultiplier`**: `2` for all (per-dumbbell weight entry, matching the convention established by the v5→v6 migration for existing dumbbell exercises).
- **`force`**: `"push"` or `"pull"` per exercise (presses/raises = push, curls/rows/pulls/hinges = pull).
- **`mechanic`**: `"compound"` for multi-joint movements (presses, rows, squats, deadlifts, lunges, RDLs, Arnold press, renegade row, squat-to-press combos); `"isolation"` for single-joint (curls, extensions, raises, flyes, shrugs, calf raises, wrist work).
- **`primaryMuscles`**: chosen so the existing `MuscleGroup.datasetMuscles` matching produces the right bucket:
  - Chest moves → `["chest"]`
  - Back rows / pullover → `["lats"]` or `["middle back"]`
  - Shoulder presses, raises, shrugs → `["shoulders"]`
  - Bicep work → `["biceps"]`; tricep work → `["triceps"]`; forearm work → `["forearms"]`
  - Quad-dominant legs (squats, lunges, step-ups, calf raise) → `["quadriceps"]`
  - Hip-hinge legs (deadlifts, RDLs, hip thrust) → `["hamstrings"]` or `["glutes"]`
  - Core (V-up) → `["abdominals"]`

Leg push/pull split (Endomondo doesn't differentiate; assignment comes from the `primaryMuscles` choice above):

- **`LEGS_PUSH`** (via `quadriceps`/`calves`): goblet squat, lunge, walking lunge, reverse lunge, side lunge, split squat, Bulgarian split squat, jumping squat, sumo squat, step-up, standing calf raise, squat-to-overhead-press, lunge-with-bicep-curl.
- **`LEGS_PULL`** (via `hamstrings`/`glutes`): deadlift, Romanian deadlift, single-leg RDL, stiff-leg deadlift, frog hip thrust.

The instructions array carries one element: the Endomondo description, lightly rewritten as needed for clarity.

### 2. Schema & DB migration

**`data/local/entity/ExerciseEntity.kt`** — remove `val images: List<String>` from the data class.

**`data/model/ExerciseJson.kt`** — remove `val images: List<String> = emptyList()`.

**`data/local/DatabaseSeeder.kt`** — remove `images = images` from the `toEntity()` mapper. The seeder's `if (dao.count() > 0) return` guard stays. Once the migration empties the exercises table, the next call to `DatabaseInitializer.initialize()` (App startup) re-seeds from the new JSON.

**`data/local/AppDatabase.kt`** — bump `version = 6` to `version = 7`. Add:

```kotlin
val MIGRATION_6_7 = object : Migration(6, 7) {
    override fun migrate(db: SupportSQLiteDatabase) {
        // Wipe per-exercise data so the new library starts clean.
        db.execSQL("DELETE FROM workout_sessions")
        db.execSQL("DELETE FROM exercise_logs")
        db.execSQL("DELETE FROM personal_records")
        db.execSQL("DELETE FROM active_workout")
        db.execSQL("DELETE FROM exercises")

        // Drop the images column. SQLite < 3.35 doesn't support ALTER TABLE DROP COLUMN,
        // so use the table-rebuild idiom.
        db.execSQL("""
            CREATE TABLE exercises_new (
                id TEXT NOT NULL PRIMARY KEY,
                name TEXT NOT NULL,
                force TEXT,
                level TEXT NOT NULL,
                mechanic TEXT,
                equipment TEXT,
                category TEXT NOT NULL,
                primaryMuscles TEXT NOT NULL,
                secondaryMuscles TEXT NOT NULL,
                instructions TEXT NOT NULL,
                volumeMultiplier INTEGER NOT NULL DEFAULT 1
            )
        """.trimIndent())
        db.execSQL("DROP TABLE exercises")
        db.execSQL("ALTER TABLE exercises_new RENAME TO exercises")
    }
}
```

The migration must be added to whatever `addMigrations(...)` chain exists where `AppDatabase` is built (the Hilt `DatabaseModule`). Recovery configs and user preferences are intentionally preserved.

Room will generate `app/schemas/com.astute.body.data.local.AppDatabase/7.json` on next build.

## Verification

1. **Build clean**: `./gradlew clean assembleDebug` succeeds with no warnings about missing assets/images, no unresolved coil imports.
2. **Unit tests**: `./gradlew test` — all suites pass. In particular: `WorkoutGeneratorTest`, `MuscleGroupScorerTest`, `HomeViewModelTest`, `ManageExercisesViewModelTest`, `ExerciseDetailViewModelTest`.
3. **Schema export**: `app/schemas/com.astute.body.data.local.AppDatabase/7.json` is generated and committed.
4. **Migration smoke**: install the new build over an existing v6 install on an emulator/device, confirm the app launches without `IllegalStateException: Migration didn't properly handle ...`.
5. **End-to-end manual test**: launch app → Home generates a workout drawing only from the new 74 — confirm no exercises like "Cable Crossover" appear → tap an exercise → Detail screen opens with no image area, instructions visible at the top → ManageExercises lists ~74 entries → start a workout, log a set, finish, confirm summary works → re-open Home and observe a fresh workout suggestion.
6. **APK size**: visibly smaller (assets folder gone).
